## Sandbox-first rule (load-bearing)

Every Unite-Group schema change, migration, or DB-writing experiment goes
through **the sandbox** before touching prod. The sandbox is an isolated
Supabase project (ref `xgqwfwqumliuguzhshwv`) whose schema is mirrored
from prod (ref `lksfwktwtmyznckodsau`) on demand.

### The wizard

`scripts/sandbox-wizard.sh` is the single entry point. Never reach for `psql`,
`supabase db push`, or the Supabase MCP `apply_migration` against prod directly.

```bash
./scripts/sandbox-wizard.sh setup              # first-time setup
./scripts/sandbox-wizard.sh sync               # re-mirror prod schema to sandbox
./scripts/sandbox-wizard.sh apply migration.sql # apply to sandbox only
./scripts/sandbox-wizard.sh diff               # schema diff prod vs sandbox
./scripts/sandbox-wizard.sh status             # health + last sync
./scripts/sandbox-wizard.sh promote migration.sql  # apply to prod (needs typed "promote to prod")
```

### Why a wizard and not Supabase branches

Prod's migration history is unreproducible (1,665 prod tables, only 32 replay
from migration files). Supabase `create_branch` fails with `MIGRATIONS_FAILED`.
The wizard sidesteps this by mirroring the live schema with `pg_dump --schema-only`.

### Prereqs

- 1Password CLI signed in: `eval $(op signin)`
- 1Password vault `Unite-Group-Infrastructure` must hold:
  - `SUPABASE_ACCESS_TOKEN` (https://supabase.com/dashboard/account/tokens)
  - `UNITE_GROUP_DB_PASSWORD` (prod DB password)
  - `UNITE_GROUP_SANDBOX_DB_PASSWORD` (sandbox DB password)
- `psql` + `pg_dump` (Postgres 17+) — `brew install postgresql@17`

`.env.sandbox` is generated by the wizard and gitignored. Never commit credentials.
