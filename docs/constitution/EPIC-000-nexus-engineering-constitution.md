## 1. Governing Position

Nexus will not operate as a collection of loosely connected agents, prompts, scripts, or third-party integrations.

Nexus will operate as a **governed adaptive control plane for organisational cognition and execution**.

The Adaptive Control Plane will:

* interpret organisational intent;
* discover existing capabilities before creating new ones;
* select capabilities based on evidence rather than popularity;
* coordinate agents, models, tools, MCP servers, CLIs, workflows, and human participants;
* enforce constitutional workflow states;
* preserve human decision authority;
* verify execution outcomes;
* measure capability performance;
* learn from every recommendation, approval, execution, override, failure, and verification result;
* improve future orchestration without silently modifying constitutional authority.

The Adaptive Control Plane is not merely a wrapper. A wrapper provides a simplified interface around another component.

The Adaptive Control Plane is not merely a harness. A harness is primarily used to execute, test, or observe components under controlled conditions.

The Adaptive Control Plane is the **governed operational intelligence layer** that connects authority, evidence, capabilities, orchestration, execution, verification, and organisational learning.

## 2. Binding Engineering Principles

The following engineering principles are effective immediately.

1. **Discover before asking.**
2. **Reuse before creating.**
3. **Capability before model.**
4. **Standards before proprietary coupling.**
5. **Evidence before opinion.**
6. **Independent challenge before recommendation.**
7. **Human authority before execution.**
8. **Verification before completion.**
9. **Observability before production deployment.**
10. **Security before convenience.**
11. **Reproducibility before acceptance.**
12. **Explicit interfaces before integration.**
13. **Structured state before conversational memory.**
14. **Measured performance before promotion.**
15. **Proven maintenance before strategic dependency.**
16. **Sandbox before trust.**
17. **Least privilege before broad access.**
18. **Build only when adoption or adaptation cannot satisfy the requirement.**
19. **Internally built capabilities must pass the same or higher standards as external capabilities.**
20. **All material technical decisions must remain explainable, reviewable, reversible, and attributable.**

No capability will be adopted because it is fashionable, rapidly generated, visually impressive, or supported by persuasive marketing.

No internally generated capability will receive preferential treatment because Nexus created it.

## 3. Definition of Engineering-Grade Material

A capability is considered engineering-grade only when sufficient evidence exists across the following areas:

clear problem definition · documented architecture · explicit interfaces · deterministic or bounded behaviour · meaningful automated tests · reproducible installation and execution · dependency transparency · secure defaults · error handling · structured logging · observability · versioning · migration strategy · rollback strategy · maintainership evidence · licence clarity · documented limitations · threat analysis · operational ownership · measurable acceptance criteria.

Code that lacks these characteristics will be classified as one of: prototype · experiment · demonstration · unverified capability · unsupported capability · abandoned capability · unsuitable for production.

The phrase **vibe coded** will not be used as the sole rejection criterion. Nexus will instead identify the observable engineering deficiencies that make the material unsuitable.

## 4. EPIC-000: Open Source and Platform Capability Intelligence

EPIC-000 establishes the permanent capability discovery, evaluation, intake, and learning mechanism used before Nexus builds or adopts technical functionality.

It covers: GitHub repositories · GitLab and other source repositories where relevant · Hugging Face models, datasets, Spaces, libraries, and evaluation assets · Model Context Protocol servers, SDKs, registries, and supporting infrastructure · OpenAI platform capabilities, SDKs, tools, agents, applications, and supported integrations · Anthropic platform capabilities, SDKs, tools, agents, MCP ecosystem resources, and supported integrations · other top-tier model providers · commercial orchestration platforms · open-source orchestration frameworks · workflow engines · identity platforms · policy engines · knowledge graph systems · vector and relational data systems · observability platforms · evaluation frameworks · agent communication standards · CLI frameworks · security tools · dependency analysis tools · software supply-chain systems · governance and approval systems.

EPIC-000 is not a one-time market scan. It is a continuous organisational capability-intelligence function.

## 5. Capability Registry Family

The original Capability Registry will become the primary entry point into a governed family of interconnected registries.

**5.1 Capability Registry** — the abstract organisational capability required, independent of implementation. Examples: authenticate a human; evaluate evidence; retrieve organisational knowledge; challenge a recommendation; execute an approved change; verify an external outcome; generate an MCP server; evaluate a software repository.

**5.2 Skills Registry** — reusable procedural skills, instructions, workflows, prompts, evaluation criteria, and operational playbooks.

**5.3 MCP Registry** — approved MCP servers, tools, transports, schemas, trust classifications, permissions, owners, versions, and test results.

**5.4 Agent Registry** — approved agent roles, responsibilities, authority boundaries, tool access, model requirements, evaluation performance, and escalation paths.

**5.5 Model Registry** — approved models and configurations: supported use cases · provider · version · cost profile · latency · context characteristics · tool-use performance · structured-output reliability · safety constraints · data-handling conditions · benchmark results · fallback relationships.

**5.6 CLI Registry** — approved command-line tools, installation methods, permissions, supported platforms, licences, versions, security findings.

**5.7 Framework Registry** — orchestration frameworks, agent frameworks, workflow engines, SDKs, libraries, infrastructure components.

**5.8 Vendor and Platform Registry** — commercial providers, contractual restrictions, service dependencies, data-processing considerations, strategic risks, exit options.

**5.9 Policy Registry** — constitutional, engineering, security, data, operational, and approval policies in machine-readable and human-readable forms.

**5.10 Evaluation Registry** — test suites, benchmarks, red-team cases, acceptance criteria, prior results, regressions, confidence levels.

**5.11 Evidence Registry** — evidence records, provenance, timestamps, source classifications, quality scores, conflicts, supporting artefacts.

**5.12 Knowledge and Decision Graph** — connects capabilities, implementations, evidence, decisions, people, agents, outcomes, failures, lessons, and organisational context. **The graph must preserve why a decision was made, not merely what was selected.**

## 6. Twenty-Stage Capability Intake and Engineering Review

Every material external or internal capability will move through the following governed states.

1. **Intent Definition** — problem, outcome, authority boundary, constraints, success criteria.
2. **Capability Decomposition** — separate the required capability from any assumed vendor, model, framework, or implementation.
3. **Registry Search** — search the Capability Registry and related registries before external discovery or new construction.
4. **Existing Internal Reuse Review** — determine whether an approved internal capability already satisfies the requirement.
5. **Ecosystem Discovery** — trusted external ecosystems: GitHub, Hugging Face, MCP resources, major model platforms, standards bodies, established commercial platforms.
6. **Candidate Normalisation** — convert candidate information into a common comparison schema.
7. **Provenance Verification** — repository ownership, package origin, release history, maintainer identity, signatures, checksums, authoritative distribution channels.
8. **Licence and Legal Review** — licence type, restrictions, copyleft implications, patent clauses, commercial-use conditions, attribution obligations, dependency licence conflicts.
9. **Security Triage** — known vulnerabilities, unsafe installation behaviour, exposed secrets, suspicious binaries, dependency risks, network behaviour, privilege requirements.
10. **Isolated Acquisition** — clone, download, or install only inside a controlled sandbox with no production credentials and restricted network access.
11. **Static Engineering Analysis** — architecture, code quality, dependency structure, configuration handling, secrets handling, test design, error handling, type safety, maintainability.
12. **Supply-Chain Analysis** — SBOM, transitive dependencies, package integrity, abandoned dependencies, build provenance.
13. **Dynamic Behaviour Analysis** — run in isolation while observing filesystem changes, process creation, network activity, resource usage, logs, errors, unexpected side effects.
14. **Functional Validation** — test against the defined organisational requirement rather than the capability's own demonstration examples.
15. **Adversarial and Failure Testing** — malformed inputs, unavailable dependencies, permission failures, timeouts, prompt injection, data poisoning, tool misuse, concurrency, recovery behaviour.
16. **Constitutional Fit Review** — does the candidate support explicit state transitions · evidence capture · independent challenge · approval boundaries · human authority · verification · auditability · finite context management · organisational learning.
17. **Comparative Engineering Score** — compare shortlisted candidates using the approved scorecard; record all supporting evidence.
18. **Independent Challenge and Judge Validation** — a separate reviewer challenges assumptions, scoring, hidden dependencies, optimistic conclusions, and proposed integration strategy. A judge capability validates the reasoning and identifies unresolved uncertainty.
19. **Board Decision** — the Board approves one of five dispositions: adopt · adapt · trial · observe · reject.
20. **Controlled Integration and Continuous Review** — integrate through explicit interfaces, run production acceptance tests, monitor real-world outcomes, record lessons, and periodically reassess as versions, maintainers, threats, or requirements change.

## 7. Engineering Scorecard

Each candidate will receive a score supported by evidence.

| Dimension | Evaluation Focus |
| --- | --- |
| Constitutional fit | Alignment with Nexus states, authority, evidence, challenge, approval, verification, and learning |
| Functional fit | Ability to satisfy the actual capability requirement |
| Architecture quality | Modularity, interface clarity, separation of concerns, extensibility |
| Security posture | Secure defaults, vulnerability history, privilege requirements, threat exposure |
| Supply-chain integrity | Dependency provenance, reproducibility, release integrity, SBOM |
| Test maturity | Unit, integration, end-to-end, regression, security, failure-path testing |
| Operational maturity | Logging, metrics, tracing, alerting, backup, recovery, runbooks |
| Documentation | Installation, architecture, API, limitations, migration, troubleshooting |
| Maintainability | Code clarity, dependency health, upgradeability, contributor accessibility |
| Community health | Maintainer activity, issue responsiveness, contributor diversity, release cadence |
| Licence suitability | Legal compatibility with intended organisational and commercial use |
| Data governance | Data handling, retention, residency, confidentiality, provider access |
| Interoperability | Standards support, MCP compatibility, APIs, SDKs, portability |
| Vendor or maintainer risk | Concentration, abandonment, acquisition, pricing, policy-change exposure |
| Performance | Latency, throughput, reliability, resource consumption, scalability |
| Cost | Acquisition, integration, operation, support, migration, exit |
| Lock-in risk | Ability to replace the capability without restructuring Nexus |
| Learning value | Ability to produce evidence and outcomes that improve future orchestration |
| Integration effort | Engineering complexity, organisational disruption, required specialist knowledge |
| Confidence | Strength, freshness, independence, completeness of supporting evidence |

**Scores will not replace judgement. Scores will make judgement inspectable.**

## 8. Required Candidate Dispositions

Every reviewed candidate must end with one explicit disposition.

* **Adopt** — use the capability substantially as provided.
* **Adapt** — use with controlled modifications or an internal compatibility layer.
* **Trial** — run a time-bounded, isolated pilot because evidence is promising but incomplete.
* **Observe** — do not integrate yet, but monitor development, maturity, or ecosystem adoption.
* **Reject** — do not use. Record the evidence, reasons, and conditions that would justify reconsideration.

## 9. Internal Build Policy

Building a new skill, agent, MCP server, CLI, framework component, or service is permitted only when: no approved capability satisfies the requirement · adaptation would create greater risk than building · strategic control is essential · the requirement is a genuine Nexus differentiator · security or governance requires internal ownership · existing options create unacceptable lock-in · **the Board approves the build decision.**

Every internal build must include: a capability definition · an architecture decision record · interface contracts · threat model · test strategy · evaluation suite · observability · versioning · release process · rollback process · owner · support expectations · deprecation process · scorecard assessment.

**Internally generated code must not enter the approved registry until it has passed the same twenty-stage review process as external code.**

## 10. Capability Factory

Nexus will eventually operate a governed Capability Factory that can generate specialised: skills · agents · MCP servers · CLI tools · connectors · evaluation suites · workflow components · policy modules · adapters · observability packages · knowledge-ingestion pipelines.

**The Capability Factory may generate implementation candidates, but it cannot approve its own output.** Generated capabilities must pass independent engineering review, challenge, judge validation, human approval, controlled execution, and verification.

## 11. Self-Learning Mechanism

Every review and integration will produce structured learning records. Nexus will learn: which discovery sources produce reliable candidates · which maintainers and ecosystems demonstrate sustained quality · which scorecard dimensions predict real production outcomes · where human reviewers regularly disagree with automated evaluations · which models perform best for particular engineering tasks · which agents produce reliable challenge · which capabilities fail under operational conditions · which integrations create hidden maintenance cost · which evidence sources become stale · which internal builds outperform adopted alternatives · which Board decisions produce the strongest verified outcomes.

**Learning may alter:** search order · candidate ranking · model selection · test depth · risk classification · recommended integration strategy · capability confidence · review frequency.

**Learning may not independently alter:** constitutional principles · human authority · approval requirements · security boundaries · Board decision rights · evidence requirements.

## 12. Governance Responsibilities

**Board** — retains final decision authority for material adoption, adaptation, rejection, strategic dependency, and production execution.

**Margo** — the executive interface through Telegram; translates executive intent into governed Nexus requests.

**Senior Project Manager** — constitutional; owns orchestration across discovery, review, engineering, dependency management, sequencing, risk, escalation, and verification.

**Engineering Review Capability** — performs structured technical analysis and produces evidence-backed findings. **It does not approve its own recommendations.**

**Independent Challenger** — searches for weaknesses, hidden assumptions, missing evidence, security gaps, operational risks, and alternative candidates.

**Judge** — validates the completeness and quality of the reasoning process and identifies unresolved disputes.

**Nexus** — owns the governed process, state, records, coordination, evidence, and learning. **Nexus does not own executive authority.**

## 16. Immediate Directive

The next engineering work will not begin by writing replacement code.

It will begin by implementing the minimum EPIC-000 research and evaluation mechanism required to perform a governed build-versus-adopt assessment for the Google Identity Foundation vertical slice.

The first output will be a source-backed capability landscape covering credible GitHub repositories, Hugging Face resources, MCP ecosystem components, major model-platform capabilities, standards, and commercial alternatives.

Each candidate will be evaluated as though reviewed by a multidisciplinary team with senior experience in: software architecture · systems engineering · cybersecurity · DevSecOps · platform engineering · AI engineering · data governance · product engineering · technical programme management · enterprise operations.

No candidate will be described as production-ready until the available evidence supports that conclusion.

No candidate will be installed into the Nexus production environment until the governed intake process authorises it.

**This block is now the binding engineering baseline for the next phase of Nexus.**
