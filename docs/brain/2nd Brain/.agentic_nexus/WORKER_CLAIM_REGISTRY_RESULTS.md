## Assumptions

- Worker IDs and machine roles used here are the ones established in the prior step and are distinct from the older `registries/workers.json` IDs by design.
- The role compatibility matrix in `worker_claim_task.py` is the "first safe mapping" from the task brief. The matrix is intentionally narrow at this stage — `command_node` cannot claim build/research tasks; `build_worker` cannot claim research/marketing/legal/compliance tasks; `research_worker` cannot claim engineering/QA tasks. Future work can broaden it once the actual fleet is known.
- `current_task_id` is the only signal used to detect "worker is already on an active task". When `current_task_id` is non-null, the worker is treated as not available for new claims. The current registry has no worker with a non-null `current_task_id`, so this code path is exercised by the test of the `paused` worker (which fails on `status` first) and the design is forward-compatible.
- The `assigned_worker_role_allowed` field in the dashboard is computed with a display-only mirror of the role matrix inside `generate_dashboard_status_feed.py`. The authoritative source is still `worker_claim_task.py`. The mirror must be kept in sync by hand. A future refactor could import the matrix from `worker_claim_task.py` to remove the duplication; this is a deliberate "do not overbuild" choice.
- New `claim_status` values (`worker_not_registered`, `worker_not_available`, `worker_role_not_allowed`, `worker_registry_missing`, `worker_registry_invalid`) are additive; no existing `claim_status` was removed or renamed. Downstream consumers that grep for the original values (`task_claimed`, `task_not_claimable`, `task_claim_failed_closed`, `task_not_found`, `task_blocked_*`) continue to work.
- The existing `task_denied_001` no longer returns `task_not_claimable` (it used to, because its status was `blocked_denied`). It now returns `task_claim_failed_closed` because its `required_agent_type` (`Local Build Worker`) is not in the role matrix and the registry check fails closed earlier. This is a strict tightening of the gate, not a loosening.
- The paused test worker is left in the registry so the paused test is reproducible. Removing it would mean the paused test can no longer be run. If a future cleanup step wants to remove it, it should be a deliberate, named operation.

## Risks found

- **Unknown agent types fail closed earlier than before.** The four original example tasks all have `required_agent_type` values from the older scope matrix, and one of them (`Local Build Worker`) is not in the new role matrix. This is the intended fail-closed behavior, but it does change the `claim_status` for `task_denied_001` from `task_not_claimable` to `task_claim_failed_closed`. Strictly an improvement; no code path was loosened.
- **Display matrix duplication.** The role matrix is now defined in both `worker_claim_task.py` (authoritative) and `generate_dashboard_status_feed.py` (display mirror). If the authoritative matrix changes, the display mirror must change too. This is documented in a comment at the top of the display mirror.
- **No automated freshness check on the registry.** A worker that goes offline without sending a heartbeat will still appear `available` in the registry. A freshness check (e.g. "if `last_heartbeat_at` is older than N minutes, treat as `offline`") is not part of this step.
- **Fixture tasks pollute the live queue.** Three new `task_worker_registry_*` tasks live in `task_queue.jsonl`. They are clearly prefixed and source-tagged as `local_worker_registry_test_fixture`, but they will appear in the dashboard summary until removed.
- **Paused test worker remains in the registry.** Same trade-off as above — reproducibility vs cleanliness. Clearly named and source-tagged so a future cleanup step can remove it safely.

## What remains missing

- The worker is still allowed to remain in a `claimed` task even though the registry says `current_task_id` is `null`. The claim gate does not currently set `current_task_id` on the worker record; the next step is to fix that.
- `worker_complete_task.py` does not yet read the registry to verify the worker is still allowed, and does not yet reset the worker's `current_task_id` and `status` on success. This is the expected next step.
- There is no "auto-fail" for the case where `worker_registry.jsonl` is deleted or corrupted at runtime. The gate currently returns `worker_registry_missing` / `worker_registry_invalid`, which is fail-closed, but a human operator must repair the registry before any further claims can succeed. A future step could add a "fail-closed with explicit operational signal" path.
- The role matrix is intentionally narrow. Once the actual three-computer fleet is online, the matrix should be reviewed and broadened to match the real capability map.
- The dashboard mirror of the role matrix is duplicated. A future refactor should import the authoritative matrix from `worker_claim_task.py`.

## Smallest useful next action

Update `worker_claim_task.py` so a successful claim sets the worker's `current_task_id` to the claimed task's `task_id` and sets the worker's `status` to `busy` in `worker_registry.jsonl`. Then update `worker_complete_task.py` so a successful completion clears `current_task_id` and sets the worker's `status` back to `available` (or to `needs_review` if the completion evidence is invalid). This closes the lifecycle: a worker can be `available` → `busy` (claimed) → `available` or `needs_review` (completed), and the registry stays in sync with the queue.

A secondary improvement: have `worker_claim_task.py` reject claims with `claim_status=worker_not_available` if the worker's `last_heartbeat_at` is older than a configurable freshness window. This would catch offline workers that forgot to send a heartbeat.
