## Commands run

```bash

cd /Users/phillmcgurk/2nd-brain/.agentic_nexus

python3 -m py_compile update_worker_heartbeat.py

python3 register_worker.py --worker-id command-node-01 --worker-name "Command Node 01" --machine-role command_node --status available
python3 register_worker.py --worker-id local-build-worker-01 --worker-name "Local Build Worker 01" --machine-role build_worker --status available
```

## Assumptions

- The `worker_id` strings (`command-node-01`, `local-build-worker-01`, `local-research-worker-01`) are distinct from the older `registries/workers.json` IDs (`command-node`, `build-worker`, `research-bi-worker`) by design. The new IDs follow the explicit names in the task brief, while the older file is preserved for reference and not auto-rewritten.
- The role-name conventions used here are the ones that appear in the task brief: `command_node`, `build_worker`, `research_worker`. The default `allowed_agent_types`, `allowed_task_types`, `allowed_projects`, `capabilities`, and `limitations` per role are derived from `agent_scope_matrix.json`, `registries/projects.json`, and `registries/agents.json`, plus a small amount of role-specific knowledge (e.g. SEO/AEO/GEO is in scope only for the research role).
- `last_heartbeat_at` is set to the registration timestamp on first registration, and refreshed to the heartbeat timestamp on every successful heartbeat. This means a worker that has never sent a heartbeat still has a `last_heartbeat_at` (its registration time), which keeps the dashboard "Worker visibility" section populated for the initial rollout.
- `current_task_id` is always `null` at this step because the next step (worker claim gate enforcement) is the one that will set it. Pre-seeding it would imply a worker is already running work, which this step must NOT do.
- The fail-closed fallback in `generate_dashboard_status_feed.py` still returns a valid JSON object when `task_queue.jsonl` is missing/unparseable. The new worker fields default to `0` and an empty list, which is the correct shape for downstream consumers.
- Re-running `register_worker.py` with the same `--worker-id` is treated as an UPDATE (in place), not a duplicate INSERT. This is verified by step 3 returning `worker_updated` and preserving `registered_at`.

## What remains missing

- The `worker_claim_task.py` script does NOT yet consult `worker_registry.jsonl`. A worker can still claim a task with any `--worker-id` string. This is the explicit next step.
- The dashboard feed does not yet include a "Worker -> Task" assignment view; it only shows counts and per-worker status.
- The `last_heartbeat_at` field is currently set to a real timestamp on registration. There is no separate "first heartbeat ever" field. If we want to distinguish "registered but never heartbeated" from "registered and heartbeated", we can add a `first_heartbeat_at` field in a later step.
- The earlier hand-written `registries/workers.json` is still present. Future work should decide whether to retire that file, migrate its data into the new `worker_registry.jsonl`, or keep both side by side with a documented mapping. The current step does not delete or rewrite it.
- There is no automated freshness check (e.g. "if last_heartbeat_at is older than X minutes, mark the worker offline"). That belongs to a later step.
- There is no machine-specific `host` or `os` field on each worker. The three workers are identified by `worker_id` only; mapping to a specific physical computer is out of scope for this step.

## Smallest useful next action

Update `worker_claim_task.py` so a worker can only claim a task if all of the following are true:

1. the supplied `--worker-id` exists in `worker_registry.jsonl`;
2. that worker's `status` is `available`;
3. that worker's `machine_role` is allowed for the task's `required_agent_type` (e.g. `build_worker` for coding/build/test/qa tasks, `research_worker` for diagnostic/obsidian/research tasks, `command_node` for coordination/dashboard/approval tasks).

Until that update lands, registered workers are visible in the dashboard but cannot actually run any task — the next registration step is purely visibility/heartbeat.
