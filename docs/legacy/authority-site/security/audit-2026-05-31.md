## FINDINGS BY SEVERITY

### CRITICAL (3)

#### C1: gitleaks pre-commit hook is inactive
- **What:** `core.hooksPath` is set to `.husky/_` but the gitleaks hook lives in `.githooks/pre-commit`
- **Also:** `gitleaks` binary is NOT installed on this machine
- **Impact:** Secret leaks in commits go undetected at the local stage. The May 10
  incident (Anthropic key pasted into chat) prompted this hook, but it's not firing.
- **Fix:**
  ```bash
  brew install gitleaks
  git config core.hooksPath .githooks
  # OR merge the gitleaks step into .husky/pre-commit
  ```
- **Action:** Developer (immediate)

#### C2: Deepsec weekly security scan failing silently
- **What:** Last 2 runs (May 18, May 25) both concluded with `failure`
- **Impact:** Automated security regression detection is blind for 2+ weeks
- **Fix:** Investigate failure logs, fix the `npx deepsec scan` invocation
- **Action:** Developer (this week)

#### C3: Branch protection too permissive
- **What:** `main` branch requires 0 approving PR reviews, admins can force-push
- **Impact:** A compromised account or accidental force-push could inject malicious code
- **Fix:**
  ```
  Required reviewers: 1
  Enforce for admins: true
  Require conversation resolution: true
  ```
- **Action:** Board decision (policy change)

### HIGH (4)

#### H1: No Content-Security-Policy header
- **What:** vercel.json sets X-Frame-Options, HSTS, nosniff — but no CSP
- **Impact:** XSS attacks can load arbitrary scripts; no defense-in-depth against injection
- **Fix:** Add a CSP header in vercel.json for the `/(.*)` source. Start with
  `report-only` mode, then enforce after validating no violations.
- **Action:** Developer (needs testing)

#### H2: 3 moderate npm vulnerabilities (postcss XSS)
- **What:** `postcss <8.5.10` has an XSS via unescaped `</style>` in CSS stringify
- **Chain:** `next@15.5.18` → `postcss@8.4.31` (vulnerable nested dep)
- **Mitigation:** Top-level postcss is 8.5.14 (safe). Risk is only if Next.js
  internally uses the vulnerable postcss for user-generated CSS output.
- **Fix:** No direct fix available without breaking changes. Monitor for Next.js patch.
- **Action:** Monitor (non-blocking, CI audit job allows moderates)

#### H3: ~40 unauthenticated API routes
- **What:** Route inventory shows many `/api/empire/*`, `/api/intelligence/*`,
  `/api/wiki/*`, `/api/internal/*` routes with no auth gate
- **Context:** Many of these appear to be read-only informational endpoints
  (health checks, public wiki, empire board-minutes)
- **Risk:** Information disclosure if any route returns sensitive data
- **Fix:** Review each unauthenticated route — confirm it's intentionally public
  or add auth. The `/api/internal/*` routes are particularly concerning.
- **Action:** Developer (route-by-route review)

#### H4: No CSRF protection on any API route
- **What:** Route inventory shows CSRF ✗ for all 69 routes
- **Mitigation:** Most routes use Bearer token auth (not cookies), which is
  inherently CSRF-resistant. The middleware auth uses Supabase session cookies
  which ARE susceptible to CSRF for logged-in users.
- **Fix:** Add CSRF tokens for cookie-authenticated routes (admin, CRM mutations)
- **Action:** Developer (scoped to cookie-auth routes only)
